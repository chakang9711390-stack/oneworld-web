# 一世界｜V1 产品研发总说明 v1

## 1. 文档目标

本文件用于把《一世界》当前最新高保真 UI 原型，整理为一套可直接进入研发阶段的统一产品说明文档。

本文档解决以下问题：

1. 一世界 V1 到底要做什么。
2. 当前 UI 原型对应的产品结构是什么。
3. 研发阶段需要围绕哪些模块、页面、数据对象与交互路径展开。
4. 哪些内容属于 V1 必做，哪些属于占位、预留或后续扩展。
5. 研发、产品、设计在进入协作前，应该用什么统一认知推进。

---

## 2. 项目背景

一世界不是单页工具，也不是单次生成器，而是一个围绕 **OPC + 工作流** 构建的生成型平台。

平台目标不是只让用户浏览内容，而是让用户能够：

- 找到适合自己的行业方向
- 找到适合自己的 OPC 路径
- 找到适合自己的工作流场景
- 逐步生成、沉淀、管理自己的项目资产
- 在“发现 → 生成 → 使用 → 留存 → 继续推进”的过程中形成闭环

当前最新 UI 原型已经明确平台的第一版主结构：

- 首页
- OPC 方案库
- 工作流库
- 我的工作台
- 登录 / 注册弹窗

因此，V1 的产品定义应以“平台基础结构可运行”为目标，而不是只做若干静态页面。

---

## 3. V1 产品定位

一世界 V1 的定位为：

> 一个面向个体经营者、内容创业者、顾问型从业者与知识工作者的 OPC / 工作流生成平台。

它的核心作用不是替代用户工作，而是帮助用户更快完成以下动作：

1. 理解自己适合什么业务路径
2. 选择或创建适合自己的 OPC
3. 选择或生成适合自己的工作流
4. 在个人工作台中承接和管理这些资产

---

## 4. 当前 V1 核心模块

### 4.1 首页

首页是整个平台的统一入口页，承担以下职责：

- 传达平台定位
- 提供搜索感入口
- 将用户分流到三条主入口：
  - OPC 方案库
  - 工作流库
  - 我的工作台

### 4.2 OPC 方案库

OPC 模块是平台的一条主线。

当前结构为：

- 先进入行业库
- 再进入某行业下的 OPC 项目卡片库
- 卡片库中包含：
  - 创建我的 OPC
  - 示例 OPC 案例卡片

该模块在 V1 的核心目标是完成“行业到 OPC 的浏览与选择框架”。

### 4.3 工作流库

工作流模块是平台的第二条主线。

当前结构为：

- 先进入行业库
- 再进入该行业下的职位卡片库
- 再进入该职位下的工作场景库
- 场景库中包含：
  - 生成我的工作流
  - 示例工作场景卡片

该模块在 V1 的核心目标是完成“行业 → 职位 → 场景”的浏览与选择框架。

### 4.4 我的工作台

工作台不是展示页，而是平台的资产承接层。

V1 阶段它至少需要承担以下职责：

- 展示当前用户已有的项目状态
- 展示当前用户已有 OPC 状态
- 展示当前用户已有工作流状态
- 为后续“继续推进项目”“继续使用工作流”“查看生成结果”预留结构

### 4.5 登录 / 注册体系

当前原型已经包含登录与注册弹窗。

V1 阶段登录系统的作用是：

- 让用户具备个人身份
- 让 OPC、工作流、项目数据具备归属关系
- 为工作台沉淀用户资产提供前置条件

---

## 5. 产品主路径

当前版本建议统一采用以下三条主路径：

### 路径一：新用户从 OPC 进入

首页 → OPC 方案库 → 行业选择 → OPC 项目卡片库 → 创建我的 OPC（预留）→ 未来进入生成/编辑流程

### 路径二：新用户从工作流进入

首页 → 工作流库 → 行业选择 → 职位卡片库 → 工作场景库 → 生成我的工作流（预留）→ 未来进入生成/编辑流程

### 路径三：老用户从工作台继续推进

首页 / 导航 → 我的工作台 → 查看已有项目 / OPC / 工作流 → 继续操作（V1 先做结构预留）

---

## 6. 当前版本的产品判断

从原型来看，当前版本已经明确了平台的信息架构，但还没有完全进入“真实业务闭环”。

因此 V1 研发阶段要先完成两件事：

### 第一件事：把平台骨架做出来

也就是把以下内容稳定落地：

- 页面结构
- 基础导航
- 模块切换
- 登录注册
- 列表浏览
- 卡片层级关系
- 工作台基础结构

### 第二件事：给后续真实生成行为预留接入口

包括但不限于：

- 创建我的 OPC
- 生成我的工作流
- 用户资产写入工作台
- 用户登录后状态关联

这意味着 V1 不一定要一次性完成所有 AI 生成逻辑，但必须把后续可接入的位置、状态流转与数据对象准备好。

---

## 7. V1 范围定义

### 7.1 V1 必做

1. 首页完整可访问
2. 顶部导航可切换
3. OPC 行业库可浏览
4. OPC 行业详情（项目卡片库）可进入
5. 工作流行业库可浏览
6. 工作流职位库可进入
7. 工作流场景库可进入
8. 我的工作台页面可进入
9. 登录 / 注册弹窗可正常打开关闭与切换
10. 整体 UI 风格、响应式与主题切换可运行

### 7.2 V1 可先占位

1. 创建我的 OPC 的真实创建逻辑
2. 生成我的工作流的真实生成逻辑
3. 搜索框真实搜索能力
4. 工作台中的真实项目读写能力
5. 第三方登录能力
6. 邮箱注册登录的真实后端联通

### 7.3 V1 不纳入首轮开发闭环的内容

1. 复杂推荐算法
2. 个性化智能推荐系统
3. OPC 深度编辑器
4. 工作流深度配置器
5. 多角色协作系统
6. 支付、商业化、订阅体系
7. 复杂运营后台

---

## 8. 与研发协作时的统一原则

### 8.1 先搭平台框架，不先追求复杂业务闭环
V1 的重点是把平台结构搭稳，而不是一开始就把所有业务生成逻辑做深。

### 8.2 组件与结构要可扩展
当前页面虽然很多内容为占位，但研发实现时要按真实平台思路拆分，避免后续全部推翻。

### 8.3 数据对象先定义清楚
即使首版只是静态或半静态页面，也要提前定义：

- 用户 User
- 行业 Industry
- OPC 分类 / OPC 项目模板
- 职位 Role
- 工作场景 Workflow Scene
- 用户工作台资产 Workspace Item

### 8.4 交互状态要预留
例如：

- 未登录 / 已登录
- 可点击 / 暂不可点击
- 空状态 / 已有数据状态
- 生成前 / 生成中 / 已生成

---

## 9. 当前原型对应的研发拆分建议

建议将 V1 按以下研发包拆分：

1. 平台框架包
   - 顶部导航
   - 路由结构
   - 通用卡片
   - 通用弹窗
   - 主题切换

2. 首页包
   - Hero 区
   - 搜索框
   - 三入口卡片

3. OPC 模块包
   - 行业分类页
   - 行业内 OPC 项目卡片页

4. 工作流模块包
   - 行业分类页
   - 职位卡片页
   - 场景卡片页

5. 工作台包
   - 指标卡片
   - 状态承接页

6. 账号体系包
   - 登录弹窗
   - 注册弹窗
   - 表单状态

---

## 10. 研发进入前的最终结论

当前这份 UI 原型已经具备进入研发准备阶段的条件。

但研发不应该只把它理解成“做一个好看的页面”，而应该理解为：

> 在一世界 V1 中，先完成平台型产品的第一层结构实现，让用户可以完成浏览、进入、切换、注册登录与资产承接的基础动作，为下一阶段接入真实 OPC / 工作流生成能力打底。

这就是当前 V1 的正确研发目标。
