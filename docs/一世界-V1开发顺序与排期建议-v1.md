# 一世界｜V1 开发顺序与排期建议 v1

## 1. 目标

本文件用于明确《一世界》V1 在实际研发推进中，应该按什么顺序开发、哪些模块可以并行、哪些模块必须前置，以及整个首轮版本的推荐排期方式。

当前原则不是“先把所有东西都做一点”，而是：

> 先把平台骨架做出来，再把数据与账号接上，最后再做联调与上线准备。

---

## 2. 推荐开发总顺序

建议整体按五个阶段推进：

### 阶段一：项目与前端骨架搭建
目标：快速把可运行 Web 壳搭出来。

包含：
- 项目初始化
- 基础目录结构
- 设计变量提取
- 主题系统
- 公共组件
- 顶部导航
- 基础布局

### 阶段二：页面结构开发
目标：把原型中的主路径页面全部真实化。

包含：
- 首页
- OPC 模块页面
- 工作流模块页面
- 工作台页面
- 登录 / 注册弹窗

### 阶段三：数据库与基础接口
目标：让前端不再只依赖静态 mock，而是具备真实数据来源。

包含：
- 数据库初始化
- 核心表结构
- 种子数据
- 查询接口
- 登录注册接口

### 阶段四：联调与状态补齐
目标：把“页面能看”推进到“页面能用”。

包含：
- 页面接真实接口
- 登录状态流转
- 工作台状态联动
- 空状态 / 禁用状态 / 默认状态补齐

### 阶段五：测试与上线准备
目标：确保 V1 能稳定对外演示和上线。

包含：
- 页面测试
- 交互测试
- 环境变量整理
- 部署准备

---

## 3. 可并行与必须串行的部分

### 3.1 必须前置
以下内容必须先完成，否则后面会卡住：

1. 项目初始化
2. 设计变量与主题体系
3. 公共组件基础层
4. 数据库基础 schema

### 3.2 可并行推进
以下内容适合并行：

1. 首页开发
2. OPC 模块开发
3. 工作流模块开发
4. 工作台页面开发
5. 登录注册弹窗开发

只要公共组件层稳定，这几块都可以并行。

### 3.3 后端可并行推进的部分
在数据库基础表建好后，可并行推进：

1. 账号接口
2. OPC 查询接口
3. 工作流查询接口
4. 工作台概览接口

---

## 4. 推荐排期（以 2 人小团队为例）

### 第 1 周
- 项目初始化
- 设计变量整理
- 公共组件与基础布局
- 首页开发
- OPC / 工作流 / 工作台路由壳开发

### 第 2 周
- OPC 模块页面完成
- 工作流模块页面完成
- 登录注册弹窗完成
- 工作台页面完成
- 数据库 schema 初版完成

### 第 3 周
- 后端查询接口完成
- 账号基础接口完成
- 前端接入真实 / 半真实数据
- 空状态 / 登录态 / 默认态补齐

### 第 4 周
- 联调修正
- 测试
- 部署
- 首版验收

---

## 5. 推荐排期（以 1 人主开发为例）

如果是一人主导开发，建议顺序改为：

1. 项目初始化 + 公共组件
2. 首页
3. OPC 模块
4. 工作流模块
5. 登录注册弹窗
6. 工作台
7. 数据库与后端接口
8. 联调
9. 部署

原因是：
- 一个人最怕频繁切上下文
- 先连续把前端页面做完，能快速看到整体效果
- 再统一接后端，效率更高

---

## 6. 排期管理原则

### 6.1 不在 V1 第一轮中加入生成逻辑深水区
像“创建我的 OPC”“生成我的工作流”这类动作，V1 第一轮只做入口预留，不要让其阻塞主版本上线。

### 6.2 页面结构优先于业务深度
如果页面结构和路由不稳定，后续所有工作都会反复返工。

### 6.3 先让主路径能走通
V1 的验收重点不是功能多，而是以下路径都能走：
- 首页 → OPC
- 首页 → 工作流
- 首页 → 工作台
- 登录 / 注册可用

---

## 7. 最终结论

一世界 V1 的正确推进节奏是：

> 基础框架 → 页面真实化 → 数据接入 → 状态补齐 → 联调上线

不要反过来，不要一上来陷入复杂生成逻辑，也不要把时间花在低优先级扩展上。
