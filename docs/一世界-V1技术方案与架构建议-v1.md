# 一世界｜V1 技术方案与架构建议 v1

## 1. 目标

本方案用于为《一世界》V1 提供可快速落地、可扩展、可继续迭代的技术实现建议。

当前阶段不追求一次性做成复杂平台，而追求：

- 能快速把最新 UI 原型落成真实 Web 产品
- 能支撑首页、OPC、工作流、工作台、登录注册等基础能力
- 能为后续接入 AI 生成能力、用户资产管理能力、推荐能力留出扩展空间

---

## 2. 技术选型建议

### 2.1 前端
建议：**Next.js（App Router）+ TypeScript + Tailwind CSS**

原因：
- 适合平台型 Web 产品快速搭建
- 路由、页面组织、组件复用成熟
- 对后续 SEO、服务端渲染、静态化支持更好
- 与 React 生态、组件生态衔接顺畅
- 后续可自然接入鉴权、服务端接口、边缘部署

### 2.2 UI 层
建议：
- Tailwind CSS 负责基础样式体系
- shadcn/ui 负责基础组件加速
- 自定义主题变量承接当前高保真原型的视觉风格

原因：
- 开发效率高
- 容易做主题扩展
- 便于研发将高保真风格组件化

### 2.3 后端
建议：**Next.js API Route / Route Handler 作为 V1 轻后端**

原因：
- V1 阶段接口量不大
- 可减少前后端分离带来的协作成本
- 便于先把账号、列表查询、工作台数据跑通
- 后续若业务变重，再拆独立服务也不晚

如果想从一开始更规整，也可用：
- NestJS 作为独立后端

但就当前阶段来说，不建议一开始把架构做重。

### 2.4 数据库
建议：**PostgreSQL**

原因：
- 结构化数据适合当前业务
- 用户、行业、模板、场景、工作台资产都适合关系型建模
- 后续支持搜索、推荐、日志、统计时扩展性更好

### 2.5 ORM
建议：**Prisma**

原因：
- 建表快
- 类型安全好
- 适合 TypeScript 团队
- 与 Next.js + PostgreSQL 组合成熟

### 2.6 鉴权
建议：
- V1 用邮箱登录 / 注册 + Session 或 JWT
- 如果追求接入速度，可直接用 **NextAuth/Auth.js**

原因：
- 当前 UI 已包含邮箱、Google、Apple 入口
- V1 可先只真接邮箱登录
- Google / Apple 保留按钮与接口预留即可

### 2.7 部署
建议：
- 前端 / 一体化应用：Vercel 或自有云主机
- 数据库：Supabase PostgreSQL / Railway PostgreSQL / 自建 PostgreSQL

如果强调国内可控部署，则建议：
- Web 应用部署到国内云主机
- PostgreSQL 独立部署
- 对象存储后续再接入 OSS / COS

---

## 3. 推荐架构形态

### V1 推荐架构

前端 + 轻后端一体化：

- Next.js Web App
- Route Handlers 提供 API
- Prisma 连接 PostgreSQL
- Auth.js 或自研轻鉴权

优点：
- 开发链路最短
- 上线最快
- 页面与接口协作成本低
- 适合当前“先搭平台骨架”的阶段

---

## 4. 模块化结构建议

建议代码层按以下结构组织：

- `app/`：页面路由
- `components/`：通用组件
- `features/home/`
- `features/opc/`
- `features/workflow/`
- `features/workspace/`
- `features/auth/`
- `lib/`：工具层
- `server/`：服务端逻辑
- `prisma/`：schema 与 migration

这样做的好处是：
- 后续模块独立扩展更容易
- 产品线不会全部混在页面目录里
- 研发更容易按模块并行推进

---

## 5. 路由建议

### 页面路由建议
- `/`：首页
- `/opc`：OPC 行业库
- `/opc/[industrySlug]`：行业下 OPC 项目卡片页
- `/workflow`：工作流行业库
- `/workflow/[industrySlug]`：行业下职位卡片页
- `/workflow/[industrySlug]/[roleSlug]`：职位下工作场景页
- `/workspace`：我的工作台

### 账号相关
- 登录 / 注册在 V1 可继续保留为弹窗
- 若后续需要独立落地页，可扩展：
  - `/login`
  - `/signup`

## 7. 首轮实现原则

### 7.1 页面优先真实化，生成能力先预留
当前阶段最重要的是让平台从原型变成可运行产品。

因此建议：
- 页面用真实路由实现
- 数据先用 mock + 数据表结合
- “创建我的 OPC”“生成我的工作流”先做占位流程

### 7.2 先让数据结构真实，哪怕交互简化
后续所有复杂能力都建立在数据结构稳定的基础上。

### 7.3 把静态原型转成可复用组件
不能直接把原型 HTML 整块搬过去，而是要拆为：
- Layout
- Nav
- Hero
- Card Grid
- Modal
- Metric Card
- Breadcrumb

---

## 8. 后续扩展兼容性

该技术方案可自然承接后续能力：

- AI 生成接口
- 搜索能力
- 个性化推荐
- 工作台真实资产管理
- 收藏 / 继续使用 / 编辑
- 商业化订阅体系
- 后台配置系统

因此 V1 不会成为一次性 Demo 架构。

---

## 9. 最终建议结论

如果目标是“尽快进入研发、快速看到真实产品、后续还能继续扩”，那么当前最优方案是：

> **Next.js + TypeScript + Tailwind + Prisma + PostgreSQL + Auth.js**

这是当前阶段成本最低、速度最快、同时还能保证后续扩展空间的方案。
