# 一世界｜基于当前网站页面框架反推的整站 PRD v1

## 1. 文档目的

本 PRD 不再从概念愿景正向发散，而是**以当前已经存在的网站页面框架、路由结构、组件形态、接口占位与数据结构为基准**，反向整理出一份可直接指导后续产品、设计、研发继续推进的整站产品需求文档。

这份文档解决 5 个问题：

1. 当前网站实际上已经做到了什么。
2. 当前页面框架对应的产品结构到底是什么。
3. 每个页面的目标、模块、交互、状态、数据依赖、边界分别是什么。
4. 哪些能力已经具备真实数据支撑，哪些还处于展示态 / 占位态。
5. 后续继续开发时，应该以什么顺序从“平台骨架”走向“真实可用平台”。

---

## 2. 当前产品判断

基于当前代码与线上结构，一世界 V1 已经不是一套纯原型，而是一个**已经具备基础路由、基础数据层、基础账号体系、基础浏览链路**的早期平台。

当前平台已经形成 3 条主线：

1. **OPC 主线**：首页 → OPC 行业库 → 行业下 OPC 项目卡片库
2. **工作流主线**：首页 → 工作流行业库 → 行业下职位卡片库 → 职位下工作场景库
3. **用户资产主线**：首页 / 顶部导航 → 我的工作台

同时，平台已具备：

- 顶部导航
- 明暗主题切换
- 登录 / 注册弹窗
- 邮箱注册 / 登录真实后端接口
- 数据库驱动的行业 / 职位 / 场景 / 项目数据读取
- “创建我的 OPC”与“生成我的工作流”的前端入口与后端占位接口

因此，当前阶段的产品定义，不应再写成“未来想做什么”，而应明确成：

> 一世界 V1 是一个围绕 OPC 与工作流双主线构建、已具备初步账号体系与数据浏览能力、正在从平台骨架过渡到真实生成闭环的生成型平台。

---

## 3. 当前版本产品定位

### 3.1 平台定位

一世界当前版本定位为：

> 面向个体经营者、知识工作者、顾问型从业者、内容创业者的 OPC / 工作流探索与生成平台。

### 3.2 核心价值

平台的核心价值不是只提供静态内容展示，而是帮助用户逐步完成以下动作：

1. 找到自己所在或想进入的行业。
2. 在该行业下找到合适的 OPC 路径或工作流角色路径。
3. 进一步找到适合自己的工作场景。
4. 在后续版本中，基于所选路径生成属于自己的 OPC 或工作流资产。
5. 将这些资产沉淀到“我的工作台”中，形成持续可管理的个人项目资产。

### 3.3 当前版本阶段定义

当前版本属于：

**平台框架已成立、数据骨架已接通、真实生成闭环尚未完成的 V1 初始可运行版。**

## 4. 当前整站信息架构

### 4.1 路由结构

当前网站实际路由结构如下：

1. `/` 首页
2. `/opc` OPC 行业库页
3. `/opc/[industrySlug]` OPC 行业下项目卡片页
4. `/workflow` 工作流行业库页
5. `/workflow/[industrySlug]` 工作流职位卡片页
6. `/workflow/[industrySlug]/[roleSlug]` 工作流场景页
7. `/workspace` 我的工作台页

### 4.2 全局能力

当前全局能力包括：

- 顶部导航
- 主题切换
- 登录按钮
- 注册按钮
- 全局弹窗式账号系统
- 通用页面容器
- 通用模块卡片
- 面包屑导航

### 4.3 三条主路径

#### 路径 A：OPC 路径
首页 → OPC 行业库 → OPC 项目卡片页 → 创建我的 OPC（当前为占位触发）

#### 路径 B：工作流路径
首页 → 工作流行业库 → 职位卡片页 → 工作场景页 → 生成我的工作流（当前为占位触发）

#### 路径 C：资产承接路径
首页 / 顶部导航 → 我的工作台 → 查看项目 / OPC / 工作流状态概览

## 5. 用户角色与典型使用场景

### 5.1 目标用户

当前版本主要面向以下几类用户：

1. 想寻找个人业务路径的个体经营者
2. 想快速进入某类行业解决方案的内容创业者 / 顾问 / 教培从业者
3. 想按岗位与工作场景调用结构化工作流的知识工作者
4. 需要沉淀自己项目、OPC、工作流资产的已注册用户

### 5.2 当前阶段主要使用场景

1. 用户第一次进入平台，通过首页快速理解平台结构。
2. 用户按行业探索 OPC 方案。
3. 用户按行业与职位探索工作流场景。
4. 用户注册或登录账号，建立个人身份。
5. 用户进入工作台查看当前资产概览。

## 6. 页面级 PRD

## 6.1 首页 `/`

### 页面目标

首页承担整个平台统一入口职责：

1. 用最短路径传达平台是“OPC / 工作流一键生成平台”。
2. 让用户快速决定进入 OPC、工作流或我的工作台。
3. 减少首页信息负担，不做重内容承载，而做结构分流。

### 核心模块

1. 顶部导航
2. Hero 标题区
3. 搜索感输入框（当前为展示型输入框视觉）
4. 开始探索按钮
5. 三张主入口卡片
   - OPC 方案库
   - 工作流库
   - 我的工作台

### 关键交互

1. 点击“开始探索”默认进入 `/opc`
2. 点击三张入口卡片分别跳转到对应模块
3. 顶部导航可进入主模块
4. 登录 / 注册按钮可打开账号弹窗
5. 主题切换可切换明暗主题

### 当前状态判断

- 首页结构完整
- 分流逻辑明确
- 搜索框目前更偏“搜索感入口”，不具备真实搜索结果能力

### 数据需求

首页当前几乎不依赖实时数据，主要依赖固定入口卡片配置。

### 后续迭代方向

1. 将搜索感输入框升级为真实搜索入口
2. 根据用户登录状态展示个性化入口
3. 增加“最近使用工作流 / 最近项目”快捷入口

## 6.2 OPC 行业库页 `/opc`

### 页面目标

让用户先按行业理解 OPC 方案库，而不是直接进入杂乱项目列表。

### 页面职责

1. 展示可浏览的 OPC 行业分类
2. 提供基础搜索 / 筛选感入口
3. 作为“行业 → OPC 项目”的第一层路径页

### 核心模块

1. 页面标题区
2. 说明文案
3. 行业搜索框
4. 行业卡片列表

### 行业卡片信息

每张卡片至少包含：

- 行业名称
- 行业说明
- 该行业下的 OPC 项目数量
- 进入链接

### 关键交互

1. 用户输入关键词后，前端即时筛选当前行业列表
2. 点击某行业卡片，进入 `/opc/[industrySlug]`

### 当前状态判断

- 已接真实数据
- 已支持前端列表筛选
- 已形成从行业到项目卡片库的跳转链路

### 数据依赖

依赖 Industry 数据，筛选条件为：

- type in [opc, both]
- status = active

并统计每个行业下的 opcProjectTemplates 数量。

### 边界说明

当前页只负责行业发现，不负责 OPC 生成，不负责项目详情页。

### 后续迭代方向

1. 增加更多行业分类
2. 增加推荐行业、热门行业、最近浏览行业
3. 接入真实搜索统计与排序逻辑

## 6.3 OPC 行业下项目卡片页 `/opc/[industrySlug]`

### 页面目标

让用户在某个行业下看到可选的 OPC 项目模板，并明确存在一个“创建我的 OPC”的个人化入口。

### 页面职责

1. 承接某个行业下的 OPC 项目模板浏览
2. 明确“模板参考”与“个性化创建入口”的区分
3. 为后续真实 OPC 创建流程预留入口位置

### 核心模块

1. 面包屑导航
2. 页面标题与说明
3. 创建我的 OPC 卡片
4. 示例 OPC 项目卡片列表

### 关键交互

1. 面包屑支持返回 OPC 行业库
2. 首卡“创建我的 OPC”点击后调用 `/api/opc/create`
3. 其余案例卡片当前为展示态

### 当前状态判断

- 页面已接真实行业与项目模板数据
- 创建入口已具备前后端接口占位
- 案例卡片当前尚未进入详情页或编辑页

### 数据依赖

依赖某行业下的 opcProjectTemplates：

- status = active
- 按 sortOrder 排序
- 区分 `isCreateEntry`

### 当前产品判断

这里本质上是“行业下模板入口页”，不是完整的 OPC 工作区。

### 后续迭代方向

1. 创建我的 OPC 接入真实创建逻辑
2. 示例卡片进入详情页
3. 用户创建后写入工作台
4. 补充 OPC 编辑 / 继续推进页面

2. 区分“生成入口”与“示例场景卡片”

4. 工作场景卡片列表

### 场景卡片信息

每张卡片至少包含：

- 场景名称
- 场景说明
- 当前卡片状态（生成入口 / 场景卡片）

### 关键交互

1. 面包屑支持返回行业页与工作流首页
2. 首卡“生成我的工作流”点击后调用 `/api/workflow/generate`

## 6.7 我的工作台页 `/workspace`

### 页面目标

承接用户的项目、OPC、工作流资产，让平台从“浏览工具”升级为“资产管理入口”。

### 页面职责

1. 展示用户当前已有 OPC 数量
2. 展示用户当前已有工作流数量
3. 展示用户当前项目数量
4. 预留后续“继续推进项目”的入口结构

### 核心模块

1. 页面头部
2. 新建项目按钮
3. 三张指标卡片
   - 当前 OPC 运行状态
   - 当前工作流运行状态
   - 项目数量

### 关键交互

1. 可从导航进入工作台
2. 新建项目按钮当前为展示型按钮
3. 指标卡片展示用户资产概览

### 当前状态判断

- 已接入数据库中的用户项目数据概览
- 已具备真实数量统计
- 但仍属于“概览页”，不是完整工作台系统

### 数据依赖

当前基于 user 的 workspaceProjects 数据统计：

- `sourceType = opc` → opcCount
- `sourceType = workflow` → workflowCount
- 全量项目数 → projectCount

### 边界说明

当前工作台不支持：

1. 项目列表展开
2. 项目详情查看
3. 项目继续编辑
4. OPC / 工作流资产详情管理

### 后续迭代方向

1. 新建项目真实化
2. 资产列表化
3. 最近访问项目
4. 继续推进入口
5. 项目详情页 / 资产详情页

## 6.8 登录 / 注册弹窗

### 页面目标

为平台建立用户身份层，并为用户资产归属、工作台承接、未来商用化接入做好准备。

### 产品职责

1. 提供邮箱注册能力
2. 提供邮箱登录能力
3. 预留 Google / Apple 登录入口
4. 以弹窗方式在全站任意页面唤起，不打断主路径

### 当前模式

- 登录与注册均为弹窗模式
- 两种模式可互相切换
- 登录和注册成功后返回成功提示

### 登录模块

包含：

1. 邮箱输入
2. 密码输入
3. 邮箱登录按钮
4. Google 登录按钮（预留）
5. Apple 登录按钮（预留）

### 注册模块

包含：

1. 昵称输入
2. 邮箱输入
3. 密码输入
4. 创建账号按钮
5. Google 注册按钮（预留）
6. Apple 注册按钮（预留）

### 关键交互

1. 登录提交调用 `/api/auth/login`
2. 注册提交调用 `/api/auth/register`
3. 成功后写入 session cookie
4. Google / Apple 当前返回“结构已预留”的提示文案
5. 弹窗支持关闭
6. 登录 / 注册支持模式切换

### 当前状态判断

- 邮箱注册 / 登录已是真实后端能力
- 第三方登录仍为前端预留态
- 登录后状态联动仍较轻，尚未在全站形成明显“已登录视图差异”

### 数据与状态要求

当前至少需要处理：

- 输入校验
- 用户是否已存在
- 密码校验
- 注册成功自动登录
- 登录成功写入会话
- 失败时错误提示

### 后续迭代方向

1. 接入真实 Google / Apple OAuth
2. 登录态全站可见
3. 工作台基于当前登录用户读取真实个人数据
4. 增加退出登录后的视图反馈

## 7. 接口与数据层 PRD

## 7.1 当前已存在接口

当前已存在的接口分为四类：

### 1）认证接口
- `POST /api/auth/register`
- `POST /api/auth/login`
- `POST /api/auth/logout`
- `GET /api/auth/me`

### 2）OPC 数据接口
- `GET /api/opc/industries`
- `GET /api/opc/industries/[industrySlug]/projects`
- `POST /api/opc/create`

### 3）工作流数据接口
- `GET /api/workflow/industries`
- `GET /api/workflow/industries/[industrySlug]/roles`
- `GET /api/workflow/roles/[roleSlug]/scenes`
- `POST /api/workflow/generate`

### 4）工作台概览接口
- `GET /api/workspace/overview`

---

## 7.2 当前数据对象

从当前代码与数据读取逻辑看，V1 当前核心数据对象至少包括：

1. User
2. Industry
3. OpcProjectTemplate
4. WorkflowRole
5. WorkflowScene
6. WorkspaceProject

---

4. 生成失败
5. 已写入工作台

通用页面需统一考虑以下状态：

1. 正常态
2. 空数据态
3. 加载态
4. 请求失败态
5. 无权限态（后续）

## 11. 本版 PRD 的最终结论

基于当前网站页面框架反推，一世界现在已经不是“要不要开始做”的阶段，而是进入了：

> **平台框架已搭好，主浏览链路已跑通，账号与数据底层已接通，下一步必须把关键入口动作和资产承接层做实。**

也就是说，后续产品工作的重心不应再停留在“继续泛化写概念 PRD”，而应转为：

1. 明确每条链路的真实闭环动作；
2. 明确生成结果如何落到用户资产；
3. 明确工作台如何成为平台留存核心；
4. 以当前页面框架为基础，持续补齐真实业务能力。

这份 PRD 应作为后续产品、设计、开发继续推进的**当前主版本产品文档**使用。
