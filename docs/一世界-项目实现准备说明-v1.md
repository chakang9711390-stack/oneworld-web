# 一世界项目实现准备说明（v1）

> 目标：把当前已完成的场景 AGENT 业务定义、资产结构、后台建模和页面原型，与 one-world 当前代码底座对齐，形成一份可以直接指导下一步研发实施的实现准备说明。

---

# 一、当前项目现状判断

当前仓库 `oneworld-web` 已具备以下基础：

## 已有基础能力
- Next.js 项目骨架已可运行
- Prisma + PostgreSQL 基础链路已打通
- 已有 Industry / WorkflowRole / WorkflowScene / OpcProjectTemplate / WorkspaceProject 等基础模型
- 已有 workflow 行业 → 职位 → 场景链路展示与 API
- 已完成基础首页、workflow 页面、API 路由与种子数据起步版
- 项目已接入 GitHub / Vercel / Neon / Prisma

## 当前缺口
当前代码里的 workflow 仍是 **偏展示型、轻结构型** 的旧模型，距离“场景 AGENT 资产系统”还有明显差距。

最关键的缺口是：
- 还没有行业上下文资产模型
- 还没有真正的 SceneDefinition / SceneExecutionConfig / SceneAuthConfig / SceneRiskRule / SceneVersion 结构
- 还没有场景后台
- 还没有 seed 场景资产真正落到代码
- 还没有为场景 AGENT 主线建立前后台联动的数据底盘

所以，接下来不是从 0 开始重做，而是：

> **在现有 workflow 骨架基础上，升级为场景 AGENT 资产系统。**

---

# 二、研发目标切分：先做什么，不做什么

## 当前阶段目标
当前阶段不是直接做 Agent 执行系统本体，而是先完成：

### 目标 1：场景资产底座
- 数据模型升级
- seed 数据落地
- 场景 API 层建立
- 前台/后台可读取同一套场景资产

### 目标 2：后台配置底座
- 能管理行业、职业、工作流、场景
- 能配置执行、权限、风控、版本、前台展示

### 目标 3：前台展示升级
- 能从“行业 × 职业 × 工作流 × 场景”读取真实资产
- 能展示标准场景 AGENT 的卡片与结构

## 当前阶段明确不做
- 不做完整 Agent 执行引擎
- 不做真实账号接管编排系统
- 不做 OPC 主线实现
- 不先做职业定制 AGENT 系统
- 不先做高风险自动执行闭环

这是为了严格遵守当前主线顺序：
**场景 AGENT → 职业定制 AGENT → OPC**

---

# 三、建议研发实施顺序

## Phase 1：数据层升级（优先级最高）

### 目标
把当前 Prisma 数据层从“旧 workflow 展示模型”升级为“场景 AGENT 资产模型”。

### 需要做的事
1. 在现有 schema 基础上新增 / 改造：
   - IndustryContext（或直接升级 Industry）
   - RoleProfile（或映射现有 WorkflowRole）
   - WorkflowTemplate（或映射现有 WorkflowScene 上游模板概念）
   - SceneDefinition
   - SceneExecutionConfig
   - SceneAuthConfig
   - SceneRiskRule
   - SceneVersion

2. 明确旧模型和新模型的关系：
   - 能复用的尽量复用
   - 不能表达新需求的就新增
   - 避免一次性把旧链路全拆坏

3. 先做一版最小可运行 migration

### 输出物
- 新版 Prisma schema
- migration
- 可运行的最小 seed

---

## Phase 2：seed 数据落地

### 目标
把已经沉淀好的首批 P0 场景真正写进代码和数据库。

### 需要做的事
1. 将首批标准场景 seed JSON 草案转为：
   - `prisma/seed-scenes.ts` 或合并进 `prisma/seed.ts`
2. 首批先灌入：
   - 电商运营 P0
   - 新媒体运营 P0
   - 招聘 P0
   - 客户成功 P0
   - 销售 P0
   - UI 设计师 P0
   - 软件工程师 P0
3. 每条场景至少包含：
   - sceneId
   - industry / role / workflow 关联
   - automationLevel
   - riskLevel
   - launchPriority
   - 工具链 / 权限 / 异常规则

### 输出物
- 可运行 seed 代码
- 本地数据库有首批场景资产

---

## Phase 3：API 层升级

### 目标
让前台和后台都能读到新场景资产。

### 需要做的事
新增 API 路由建议：
- `GET /api/scenes`
- `GET /api/scenes/:sceneId`
- `GET /api/industries`
- `GET /api/industries/:slug/roles`
- `GET /api/roles/:slug/workflows`
- `GET /api/workflows/:slug/scenes`
- `GET /api/admin/scenes`
- `POST /api/admin/scenes`
- `PATCH /api/admin/scenes/:sceneId`

### 输出物
- 场景前台读取接口
- 场景后台管理接口（可先只做读 + 本地编辑）

---

## Phase 4：前台展示升级

### 目标
把现有 workflow 页从旧的“行业 → 职位 → 场景”展示，升级为基于新资产模型的展示入口。

### 建议前台迭代顺序
1. Workflow 首页继续保留，但底层数据改走新模型
2. 行业页展示该行业下职业与工作流概览
3. 职业页展示工作流清单
4. 工作流页展示场景卡片
5. 场景详情页展示：
   - 场景价值
   - 适用对象
   - 输入输出
   - 风险等级
   - 自动化程度
   - 是否可直接购买 / 是否需定制化

### 输出物
- 面向用户的标准场景 AGENT 浏览链路

---

## Phase 5：后台第一版

### 目标
先做最小可用后台，而不是一上来做完整复杂 CMS。

### 第一版后台建议只做
1. 场景列表页
2. 场景详情页
3. 场景创建/编辑页
4. 行业列表页
5. 职业列表页
6. 工作流列表页

### 第一版可延后
- 复杂版本管理
- 批量导入导出
- 多角色权限管理
- 定制化项目后台

### 输出物
- 内部可维护的场景资产后台

# 四、与当前代码的兼容建议

## 1. 不建议暴力推倒重来
当前仓库已经上线，有现成 API 和页面链路。

所以建议：
- 保留现有 workflow 路径结构
- 底层逐步切到新模型
- 用兼容层过渡旧字段到新字段

## 2. 现有模型可作为过渡映射
例如：
- `Industry` 先升级或映射到 `IndustryContext`
- `WorkflowRole` 先映射到 `RoleProfile`
- `WorkflowScene` 先映射到部分 `SceneDefinition` 或作为旧数据导入源

## 3. 第一阶段不强求一次改完全部页面
先让：
- schema
- seed
- API
- 场景详情
跑通，再逐步替换旧前台数据源。

# 六、当前阶段的“完成定义”

如果满足以下条件，就可以认为一世界已经从“文档定义阶段”进入“实现阶段”：

1. Prisma schema 升级完成
